# VAL ARK PATTERNS BRIEF: Consumer-Appliance Architecture

Synthesis of 8 research tracks (Synology/QNAP, TrueNAS/unRAID, Home Assistant, Umbrel/CasaOS, consumer routers, Pi-hole/Nextcloud, health/fitness apps, appliance recovery UX) into direct, opinionated recommendations for Val Ark â€” an offline, self-replicating knowledge/AI/comms box a non-technical owner must commission and run like a router app.

The single hardest constraint that separates Val Ark from every product studied: **there is no cloud, no email reset, no phone-home.** Every path â€” setup, recovery, notifications, self-replication â€” must close entirely on the LAN or the physical box. Umbrel and CasaOS both fail their non-technical owners at exactly this point (SSH-and-delete-a-DB-file is the only password recovery). Val Ark must not.

---

## 1. FIRST-BOOT COMMISSIONING WIZARD

**Reach it the way every good appliance does: one memorable name, zero installed software.** Advertise `valark.local` over mDNS/Bonjour plus a fixed fallback IP; on an un-owned box the web root (server.js) redirects ALL traffic captive-portal-style to the setup wizard, not the catalog. Bare-IP hits redirect to the hostname (Pi-hole's "no SID on IP" footgun proves silent auth-fail is real). Windows fallback: plain `http://valark`. Never require a Qfinder/Web-Assistant-style downloaded finder utility â€” that is a step Synology/QNAP owners get stuck on and Val Ark can skip.

**The physical console is a first-class surface, not an afterthought.** If a monitor is attached, the boot banner (TrueNAS pattern) permanently shows: `Val Ark ready â†’ http://valark.local  (or http://192.168.x.x)` plus `Press Enter for recovery menu`. This is how a person who "can't find the IP" still gets in.

**The wizard is a short linear flow, one decision per screen, every screen skippable to a sensible default.** Recommended exact sequence:

1. **Preparing Val Ark** â€” a labeled progress screen with payload size while the box comes up and lays down the seed (Home Assistant's "Preparingâ€¦ ~700 MB", never a blank spinner). This screen sets the expectation that Val Ark downloads things and shows progress.
2. **Claim the box** â€” enter the one-time claim token printed on the physical sticker / shown on the attached console. This closes the Portainer takeover hole (an un-owned admin surface on a LAN is a real attack target, GHSA-x626-fcwx-f5pc). Until an admin exists, bind the setup surface behind this token or to console/localhost. **Fail closed, not open.**
3. **Create the admin** â€” the ONE mandatory decision: admin name + password, with a live plain-language strength meter (Nextcloud) that nudges but does not block. Offer "generate a strong one for me" (eero's random-default philosophy) and show it once.
4. **Name your box** â€” optional, pre-filled `val-ark` (used as mDNS name + mesh label). One field, skippable.
5. **Found your disk** â€” auto-detect the largest mount (valark-env.sh already does this) and present as a single confirm: **"Found a 6 TB disk â€” use all of it? [Recommended]"**. Everything about VAL_ARK_DATA paths, symlinks, caps, NFS is hidden. This is TrueNAS's Pool Wizard collapsed to one confirm and QNAP's "Max" default.
6. **What is this box for?** â€” health/fitness-app preset instead of config (Whoop's goal presets). Two plain questions: *"Who's it for?"* (Just me / My family / My community) and *"What matters most?"* (Knowledge & Wikipedia / AI chat / Messaging & comms). The librarian translates the answer into fill priorities. Default = "Fill it with everything (Recommended)."
7. **Save your recovery card** â€” generate and OFFER TO PRINT/EXPORT a recovery code as QR + printable card (invert Umbrel's "you can never see this again" trap â€” offline owners need a paper path). Blunt warning, HA-style: *"Keep this by the box. It's the only way back in if you forget your password. Val Ark is offline â€” there is no email reset."*
8. **Certificate heads-up** â€” one sentence pre-empting the self-signed-TLS browser warning (Pi-hole auto-generates the cert): *"Your browser will warn about security the first time â€” that's normal for a private offline box. Click Advanced â†’ Proceed."* Handle the scary moment before it happens.
9. **Done â†’ Go to Dashboard.** The librarian fill has already started in the background.

**Rules baked in:** every choice has a Recommended default pre-selected so the owner can click straight through; the only thing they MUST invent is a password (and even that can be generated); the wizard is relaunchable later from Settings (unRAID); a Summary + explicit "this erases the disk" gate appears only before any destructive format (QNAP's Initialize pattern), never during normal setup.

## 2. ADMIN CONSOLE INFORMATION ARCHITECTURE

**Cap the top level at 4 flat, task-labeled tabs.** Reject NAS-style desktop metaphors and 10-tab controllers â€” the winning consumer pattern is eero's Home/Activity/Settings and Oura's Now/Vitals/Trends. Recommended:

- **Home** â€” the answer to "Is my Ark OK?" One large green/yellow/red status light + one sentence, then the handful of things needing attention, then "what's downloading right now." (Â§6, Â§7.)
- **Library** â€” everything stored on the box, browsable, grouped by plain category: Wikipedia & Knowledge, AI Models, Software Mirror, Comms. This is the existing catalog with Home-Assistant-add-on ergonomics per item (Info / Logs, plus toggles for Start-on-boot, Watchdog/auto-restart, Auto-update). Tiles show status and flip to "Open" when ready (Umbrel).
- **Activity** â€” the download/task queue and the self-heal event feed (Â§5, Â§6).
- **Settings** â€” ONE consolidated panel (Synology Control Panel principle: one place to look, never scattered): box name, admin & recovery, network/services, mesh. Visible only to admin.

**Progressive disclosure is the core discipline, three tiers, each on its own screen** (Whoop): Tier 1 = the status light + one sentence; Tier 2 = per-area cards (Storage, Downloads, Services, Content, Fleet); Tier 3 = raw logs, integrity reports, aria2 detail for the curious â€” "there but invisible."

**Basic/Expert toggle, Basic by default** (Pi-hole v6). Footprint caps (VALARK_MAX_GB / VALARK_MODEL_MAX_GB), NFS export, extra listen ports, tool-refresh cadence, eviction policy, cron internals â€” all live behind an Expert flip or an "Advanced" expander. Same screens, more knobs.

**Hide what doesn't apply yet** (unRAID hides VMs; Whoop unlocks features on day 5/7). Do not show Fleet/Mesh UI until a second node exists; do not surface eviction controls until the disk is meaningfully full. Day 1 is a single box quietly filling.

**Persistent status is always visible** â€” pin a bottom status bar (unRAID) or top health line on every page (eero) showing health + active download + free space, so the owner never navigates to know the box is fine.

## 3. ACCESS-CONTROL & IDENTITY (concrete recommendation)

**Recommended model: open content on the LAN, one admin gate for management, optional read-only viewer. Two roles, not five.**

| Surface | Access | Rationale |
|---|---|---|
| **Content** (Wikipedia/Kiwix, AI chat, comms, software downloads) | **OPEN to anyone on the LAN, no login** | The box's whole purpose is a household/community appliance. eero's guest network and Pi-hole's public read-only stats prove people should never hit a login wall to *use* or *check* the thing. A family member browsing Wikipedia is the happy path. |
| **Read-only status** (health, disk %, what's downloading, last self-heal) | **OPEN, no login** | Pi-hole exposes stats without auth; "is it working?" must be answerable by anyone. |
| **Management** (fill, evict, service toggles, settings, add a node, recovery) | **Single admin password** (the Owner created at first boot; undeletable) | Umbrel/Pi-hole single-admin reality. One gated place for control. |
| **Optional: household admin help** | **One "View-Only" second role** if ever needed | UniFi's View-Only, HA's non-admin. Do not build further. |

**Localhost/console is implicitly trusted** â€” the physical console and local recovery commands need no web login; physical presence IS the credential (Pi-hole, Home Assistant's insistence that owner-reset happens on the real console, not SSH).

**Never ship a usable default credential** (TrueNAS disables root; QNAP disables built-in `admin`). The only default is the one-time claim token, which is consumed at setup.

**Explicitly reject:** UniFi's 5-role hierarchy, Nextcloud Group-Admins, full RBAC. A home appliance owner needs exactly one admin and maybe one viewer.

**Tradeoffs, stated honestly:**
- *Open content = anyone on your LAN can read/use everything.* Correct for a knowledge appliance; the LAN boundary is the trust boundary, exactly like a router's Wi-Fi. If the owner wants isolation, that's a network-level guest-SSID concern, not a Val Ark account system.
- *Single admin = whoever has the password controls the box.* Mitigated by the physical recovery card and console rescue, not by adding roles.
- *Physical access = full control (the honest security tradeoff unRAID states plainly).* Accepted deliberately: an offline box in a closet cannot depend on cloud identity, so console presence must be able to reset it. Document this rather than pretend otherwise.

Offer optional 2FA on the admin login for the security-minded (Umbrel), strictly opt-in, web-UI only.

## 4. RECOVERY (forgot-password, lockout, factory-reset)

This is the section where the studied consumer products are weakest and Val Ark must be strongest. **Every path works with only {the box + a LAN browser} or {the box + a keyboard}. Zero internet, zero email, zero GitHub, zero account.**

**Two-tier model, borrowed from Synology's Mode 1 / Mode 2, made impossible to confuse:**

- **Tier 1 â€” Reset admin credential + network settings ONLY. Content untouched.** This is the everyday "I forgot my password" path and must never risk the multi-TB library.
- **Tier 2 â€” Full config/state factory reset.** Wipes admin + settings + services state. **Still preserves the content/model volume.**

**The non-negotiable invariant:** the tiny config/state (`valark/state`) is physically and logically separate from the multi-TB content/model mirror. NO reset, ever, wipes Wikipedia or the AI models. Both NAS vendors preserve user data across even a factory reset; Val Ark must too â€” a "panic reset" must never re-trigger a multi-day re-download. Only an explicit, Summary-gated, type-the-words disk-erase (QNAP Initialize) touches content.

**Headless box (no monitor):**
1. *Web forgot-password:* at `valark.local`, a "Forgot password?" link takes the serial/recovery code from the printed card + one security answer set during onboarding, and lets the owner set a new password (Orbi's exact local-only model). **Auto-surface this flow after 3 failed logins** (Orbi's 3-strikes).
2. *Physical button, two-tier with AUDIBLE feedback:* SHORT press â†’ **one beep** = Tier 1 (reset admin + network). LONG press â†’ **three beeps** = Tier 2 (factory config reset, content kept). Different beep counts are what stop a panicked owner from nuking a 20 TB library while trying to reset a password (Synology's explicit reasoning). After Tier 1, next web login lands straight on "set a new admin password" with no password required to reach it (Synology blank-admin flow).
3. *Console command over the trusted local path* (Tailscale-SSH is already in the fleet): `valark setpassword` (Pi-hole's exact model) and `valark auth list` to recover a forgotten username (HA's `auth list`). The reset PRINTS the new/temporary credential to the console.

**Box with a monitor:**
- The boot banner always shows `Press Enter for recovery menu` (TrueNAS Console Setup). The menu offers, in plain words: **"Reset admin password"**, **"Re-run first-time setup"**, **"Factory reset (keeps all your Wikipedia & AI content)"**. No SSH, no doc lookup â€” the recovery instructions are on the screen in front of the person (TrueNAS banner discoverability).

**Safe Mode / recovery web page** (Home Assistant): if config is fatally broken, boot core-only â€” just the web UI + recovery, skipping community services and the librarian fill â€” so one broken add-on can never black-screen the owner. Surface it as *"Val Ark is in recovery mode â€” repairing content, X of Y verified"* with an admin login and buttons: Restore / Reset admin / Continue. loop.sh already repairs symlinks and re-mirrors; this exposes that as a friendly recovery UI instead of a dead port.

**Discoverability is the whole game:** ship a physical recovery card with the box (the router-login-sticker + Umbrel-seed-card equivalent) listing the `.local` URL, the fallback IP, the one-time claim/recovery code, and the button/beep instructions. A person who "barely knows what GitHub is" must always have a paper path back in.

## 5. DOWNLOADS & MONITORING

**Picking priorities is a preset, never the priority model.** The owner chose an intent at first boot (Â§1 step 6); the librarian's diversityâ†’small-valueâ†’fillâ†’evict logic runs invisibly to satisfy it. The default everywhere a choice exists is **"Fill the disk with everything (Recommended)"** â€” one tap, background download/format (TrueNAS Pool Wizard, CasaOS default-install). Advanced users get a "Let me pick categories" path and Expert-mode overrides (specific mirrors, caps, evict); the 90%-case works untouched.

**The download queue IS Val Ark's marquee monitoring surface** â€” treat it like a firmware/update queue crossed with a fitness feed:
- **Async and non-blocking** (Umbrel: "you can navigate away"). Progress renders on the Library tile itself AND in the Activity queue; the tile flips to **"Open"** the moment a ZIM/model is usable.
- **Per-item plain-language cards:** name, size, percent, ETA, state (queued / downloading / verifying / done). *"Downloading now: Simple English Wikipedia â€” 2.1 GB of 4.8 GB."* Priority tier shown as a human label, not "diversity-slice."
- **Overall disk-fill %** and live headroom against caps, always visible (both NAS vendors, Umbrel Settings).
- **Resumability made visible and reassuring:** an unmistakable *"Resumes automatically after power loss"* indicator (aria2's resumable/atomic downloads already provide the data). A multi-day fill must feel monitored, not frozen.
- **Retry, not raw error, on failure** (Nextcloud updater): a single Retry button; close-and-reopen returns to the same progress.
- **Schedule heavy fills to off-peak/overnight** (eero OTA windows), optional.

**Stream it** â€” SSE already exists in server.js; push progress into a simple animated list refreshed live (eero's ~5 s Activity refresh feel).

**Updates are reassurance, not chores** (eero auto-OTA): the weekly tool refresh + live catalog refresh run automatically and invisibly; surface only *"Everything's up to date â€” last checked 2 hours ago"* with an optional "Update now."

## 6. ERROR & SELF-HEAL UX

**Self-heal FIRST, then surface only what couldn't be auto-fixed â€” but ALWAYS show what was healed.** loop.sh/verify.sh already repair symlinks, restart the server, top-up fills, re-verify. That autonomy is Val Ark's core value proposition, so make it visible and reassuring (QNAP shows auto-rebuild; Synology shows Security Advisor; Whoop silently auto-detects your Spin class).

**A Repairs/Health page of plain-language cards** (Home Assistant Repairs + Nextcloud "Security & setup warnings"), pinned at the TOP of Home, advisory not blocking so the box stays usable:
- Green reassurance: *"Wikipedia is being served âś“"*, *"Everything healthy â€” last self-check 2 min ago."*
- Calm auto-fix reports (never a stack trace): *"Web server was down â€” restarted it automatically. Nothing to do."* *"Re-linked 3 files."* *"Catalog refreshed."* â€” the Whoop-auto-detects-Spin tone.
- Actionable items: **what happened + the ONE recommended next step + one button** (QNAP: "Disk drive X failed or is unplugged. Please check the summary page."). *"Disk full â€” evicting low-priority content automatically."* *"Drive health warning â€” [Check drive]."*
- Badge count on the nav (HA/TrueNAS); severity grammar critical/warning/info with Dismiss / Dismiss-All and dismissals that persist across reboots.

**Attribute faults to the correct layer** (eero's best trick â€” a red WAN LED is "your ISP, not the box"). *"The internet is down â€” that's your provider. Val Ark is fine and still serving Wikipedia offline."* This single behavior stops a non-expert from factory-resetting a working box. Likewise render known-benign transients (the NodeBB `/app/forum/` 503 burst quirk) as *"recovering,"* never "ERROR."

**Notifications are offline-native.** No cloud push, no email. One aggregating bell/inbox (QNAP Notification Center, TrueNAS alert bell) with severity filters (All / Critical / Warning / Info / Dismissed) and frequency control (Immediately / Daily / Never â€” never nag). Delivery routes into the box's own comms channels (local mail/IRC/message board it already ships) plus an optional on-box LED and status line.

**Strict color grammar everywhere** (Whoop/Oura): green = healthy, yellow = attention, red = act. No decorative accents â€” a glance at any card in the whole UI means the same thing. If the hardware has an LED, drive it with eero's vocabulary (solid = healthy, blinking = working/updating, red = problem) and make the web banner color MATCH the LED so on-device and in-app never disagree.

**"Learn more" links resolve to bundled offline docs / a ZIM page â€” never a dead internet URL.** Val Ark already mirrors Kiwix content; point there.

**Eliminate credential hunting** (Umbrel): any tool that generates a login shows it in a post-install dialog and keeps it retrievable via right-click / "Show credentials" on the tile. The owner never opens a log to find a password.

## 7. SIMPLICITY PRINCIPLES (the top rules)

1. **One memorable local name, zero installed software, zero typed IPs.** `valark.local` + captive-portal first-run redirect + printed fallback IP. Copy find.synology.com's discovery but offline; skip the finder utility entirely.
2. **One big status light answers "Is my Ark OK?" before any detail.** Whoop's arm's-length Recovery score / eero's one health line. Green = healthy and filling 24/7; yellow = attention; red = act. Everything else is a tap away.
3. **The only mandatory setup decision is the admin password** (and even that can be auto-generated). Auto-detect the disk, auto-start the fill, default every other choice. Frame "no account, no internet needed" as the feature â€” never nag toward a cloud account you can't offer.
4. **Recommended defaults the owner just accepts** (Google Fit leaning on WHO/AHA; TrueNAS pre-checking "Encryption: None"). The owner confirms a sensible policy; they never invent one. "Fill the disk, stay healthy, keep content fresh" is THE Recommended default.
5. **Progressive disclosure: three tiers, Basic mode default, hide what doesn't apply yet.** loop.sh/verify.sh internals, caps, NFS, eviction stay invisible until an Expert flip or until there's something for them to act on (Whoop day-5 unlock, unRAID hiding VMs).
6. **Split viewing from managing.** Open content on the LAN; one admin gate for control. Never make anyone log in just to read Wikipedia or check health.
7. **The box visibly takes care of itself.** Tell the owner *"This appliance checks and repairs itself every 30 minutes"* and show the repairs as calm "Fixed automatically" cards. Self-heal is the product; surfacing it is the trust.
8. **Every error = what happened + one next step, in plain words + one button.** No stack traces, no exit codes on the primary surface (behind an "Advanced" expander only). Avoid Whoop's failure mode â€” jargon that leaves the user unsure what to DO.
9. **Never let recovery or reset endanger content.** State/config separate from the multi-TB library, always. Panic-reset must be safe.
10. **Recovery is a first-class, fully-local, paper-backed flow â€” not an afterthought.** Because there's no email reset, the printed recovery card + console menu + two-tier button-with-beeps is the offline equivalent of a router's login sticker. This is the single most important thing to get right for a box in a closet owned by someone who "barely knows what GitHub is."
